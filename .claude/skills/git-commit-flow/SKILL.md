# git コミット実行フロー

## 前提(絶対厳守)

- **ユーザーが明示的に「コミットして」と依頼したときだけ**実行する。指示されていないのに勝手にコミットしない
- `-A` / `.` でのステージングは禁止。関連ファイルを個別に指定する(`.env` 等の機密ファイルや意図しない大きいバイナリの混入を防ぐため)
- `--no-verify` / `--no-gpg-sign` は使わない(明示的に指示された場合を除く)
- 既存コミットへの `--amend` は、ユーザーが明示的に要求した場合のみ。新しいコミットを作るのが基本
- pre-commitフックが失敗したら、原因を直してから**新しいコミット**を作る(失敗した時点でコミットは成立していないため、amendは直前のコミットを壊す)

## 手順

1. 次を並列で実行し状況を把握する
   - `git status`(`-uall` は使わない。大規模リポジトリでメモリ問題になりうるため)
   - `git diff`(ステージ済み/未ステージ両方の変更内容)
   - `git log --oneline -10`(このリポジトリの直近のコミットメッセージの書き方を確認)

2. 変更内容を分析し、CLAUDE.mdのプレフィックス表に沿ってコミットメッセージを起草する

   | プレフィックス | 用途 |
   |--------------|------|
   | feat | 新機能の追加 |
   | fix | バグの修正 |
   | docs | ドキュメントの変更 |
   | style | フォーマット変更(動作に影響しない) |
   | refactor | リファクタリング(機能変更なし) |
   | test | テストの追加・修正 |
   | chore | ビルド設定・ツール関連の変更 |

   メッセージは「何を」ではなく「なぜ」を中心に1〜2文で簡潔にまとめる

3. 機密情報(`.env`、credentials等)が含まれていないか確認する。含まれていればユーザーに警告し、そのファイルは除外する

4. 関連ファイルを個別に `git add` し、ヒアドキュメントでコミットメッセージを渡してコミットする

   ```bash
   git add <file1> <file2>
   git commit -m "$(cat <<'EOF'
   <prefix>: <変更内容の要約>

   Co-Authored-By: Claude Sonnet 5 <noreply@anthropic.com>
   EOF
   )"
   ```

5. `git status` を実行し、コミットが成立したことを確認する

## 注意

- リモートへの `git push` はこのSkillの範囲外。別途明示的な指示があるまで行わない
- コミット対象に生成ファイル(`*.freezed.dart`, `*.g.dart`, `router.g.dart`)が含まれる場合、それらは通常ソース側の変更と一緒にコミットして問題ない(生成ファイル自体を手編集していない限り)
